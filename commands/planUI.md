# /planUI — CoreMind UI Pipeline

Specialization of `/plan` for UI and design work. Instead of routing through every domain, this command orchestrates the full UI surface — 26 skills, 3 agents, 3 MCPs, 5 sub-commands — around a design-workflow pipeline.

> **When to use `/planUI` vs `/plan`**: use `/planUI` for pure UI/design tasks (build a page, generate a design system, audit accessibility). Use `/plan` when UI is one piece of a cross-domain task (e.g., "build the feature + its UI + its API + its tests"). Use `/ultraplan` for enterprise-scale cross-system UI work with governance requirements.

## The Pipeline — 8 Stages

### Stage 1: BRIEF — understand the human and the job

Before any file is touched, answer in concrete terms (adapt from `interface-design`):

- **Who is this human?** A teacher at 7am with coffee? A developer debugging at midnight? A C-suite executive reviewing KPIs? Specifics.
- **What verb must they accomplish?** "Grade submissions." "Find the broken deployment." "Approve the payment." Not "use the dashboard."
- **What should this feel like?** Words with meaning. *"Warm like a notebook"*, *"cold like a terminal"*, *"dense like a trading floor"*, *"hushed like a late-night atelier"*. Not "clean" or "modern."

If you can't answer these, stop and ask the user. **Do not guess. Do not default.**

**Surface type** — narrow the route:

| Surface | Primary skill | Avoid |
|---------|---------------|-------|
| Dashboard / admin / tool | `interface-design` | react-bits heavy backgrounds; keep restraint |
| Marketing / landing / hero | `impeccable-design` + `react-bits` | Over-dense data UIs |
| Design-system generation | `ui-design-system` or `hue` | Ad-hoc tokens |
| Brand documentation / application | `brand-guidelines` + brand child skill | Generic Tailwind |
| Reference extraction | `design-inspector` or `clone-website` | Guessing at tokens |
| Video UI | `video-ui-patterns` + `remotion-animation` | Web-oriented libs |

### Stage 2: ROUTE — pick the directional path

One of three paths. Pick before doing anything else.

**A) Brand-first** — user has or wants a brand identity → drive everything from the brand.
```
Flow: hue (generate brand child skill)
   → brand child skill (tokens + hero + icon kit)
   → interface-design-init (build surfaces)
   → impeccable-audit → impeccable-polish
```

**B) System-first** — user has constraints (colors, type, spacing) but no formal brand → systematic tokens then components.
```
Flow: ui-design-system (tokens, palettes, scales, architecture)
   → interface-design-init (build)
   → react-bits (cherry-pick animated components)
   → impeccable-audit → impeccable-polish
```

**C) Component-first** — user wants a specific pattern, page, or interaction → go straight to components.
```
Flow: react-bits (pick components) OR impeccable-design (generate)
   → responsive-design + framer-motion-patterns (refine)
   → baseline-ui (sanity check)
   → impeccable-polish (ship)
```

### Stage 3: DIRECTION — establish the feel before code

State the intent AND the technical approach (from `interface-design-init`):

```
Intent: [who, what verb, how it should feel]
Palette: [foundation + accent — WHY these colors fit]
Depth: [borders / subtle shadows / layered — WHY]
Surfaces: [elevation scale — WHY this temperature]
Typography: [typeface choice — WHY it fits the intent]
Spacing: [base unit — 4px / 6px / 8px]
Motion: [restrained / playful / physics-heavy — WHY]
Hero preset (marketing only): [luminous-on-gradient / device-on-mesh / sculptural-field / typographic-hero]
```

**Every choice must be explainable.** If the answer is "it's common" or "it works" — you haven't chosen, you've defaulted.

**The test**: if another AI given a similar prompt would produce the same output, you have failed. The interface must emerge from THIS user, THIS problem, THIS intent.

Reference the 17 Hue example brands (`~/.claude/skills/hue/examples/`) for inspiration:
- `atlas` — paper-ivory, navy ink, rust accent, IBM Plex Serif — data-heavy, chart-like
- `velvet` — noir editorial, Cormorant Garamond italic, champagne on near-black — hushed
- `halcyon` — cool teal sculptural glass bars, dark mode, architectural

### Stage 4: SYSTEM — generate or load tokens

**If brand-first** → invoke `hue`:
```
"make a design skill from <brand url / name / screenshot>"
```
Hue writes `~/.claude/skills/<brand>/` with 95 tokens + 40 components + hero stage + icon kit.

**If system-first** → invoke `ui-design-system`:
- Generate design tokens
- Calculate typography scale (modular scale or major-third/perfect-fifth)
- Build spacing scale (4pt or 8pt grid)
- Define component architecture
- Prepare dev handoff (CSS vars, SCSS, Tailwind config)

**If component-first** → skip to Stage 5 with inferred tokens from context.

**Token taxonomy reference**: `~/.claude/skills/hue/references/tokens-template.md` — neutral + brand + status ramps at 50–950 steps, paired with semantic mapping. Novel concepts: hero-stage recipes, icon-kit 5-attribute matching.

### Stage 5: BUILD — assemble the surfaces

**Dashboard / tool / app** (dense, restrained):
1. `/interface-design-init` — establish project `system.md`
2. Plain Tailwind + shadcn/ui primitives
3. Sparingly pull from `react-bits` for hero section, nav, or data viz
4. `responsive-design` for breakpoints
5. `framer-motion-patterns` for selective motion (max 2-3 animated elements)

**Marketing / landing / hero** (distinctive, animated):
1. `impeccable-design` — generate the creative scaffold
2. `react-bits` — pick ONE animated background + ONE headline treatment + ONE supporting effect (see `react-bits` routing tables)
3. `responsive-design` — mobile-first breakpoints
4. `market-landing` — CRO section audit (hero → value prop → social proof → CTA → pricing → FAQ)

**Video UI**:
1. `video-ui-patterns` — reusable component library
2. `remotion-animation` — spring physics, timing, composition

**For Next.js projects using `react-bits` 3D/shader/face-api components**: wrap in `'use client'` or `next/dynamic({ ssr: false })`.

### Stage 6: AUDIT — technical quality gates

Run in order. Don't skip.

1. **`impeccable-audit`** — accessibility + performance + theming + responsive + anti-patterns (scored)
2. **`baseline-ui`** — animation durations sane, type scale enforced, component a11y, layout anti-patterns
3. **`fixing-accessibility`** — ARIA, keyboard nav, focus management, contrast (WCAG 2.2 AA minimum)
4. **`fixing-motion-performance`** — compositor properties, scroll-linked motion, blur budget
5. **`fixing-metadata`** (marketing surfaces only) — titles, meta desc, OG, canonical, favicon
6. **`/interface-design-audit`** — check against project's `system.md` for token/spacing/pattern violations

Record findings. Fix P0/P1 before ship.

### Stage 7: POLISH — final micro-detail pass

Run `impeccable-polish`:
- Alignment — pixel-perfect component edges
- Spacing — consistent rhythm, no orphan gaps
- Consistency — same component renders identically across surfaces
- Micro-details — hover states, focus rings, disabled states, loading skeletons
- `/interface-design-critique` — one more pass: "where did I default? where could this be more THIS brand?"

### Stage 8: SHIP — verify + deliver

- **Evidence**: screenshots at desktop + tablet + mobile; Lighthouse scores; accessibility audit output
- **Works offline if applicable** (PWA-ready?)
- **Reduced-motion respected** (`prefers-reduced-motion` media query)
- **Dark-mode parity** if brand has both (verify every component renders correctly in both)
- **Tests**: visual regression snapshot if tooling exists; webapp-testing smoke flow for critical paths
- Save patterns back: `"Want me to save these patterns to .interface-design/system.md?"` (from interface-design-init flow)

## Typed-clarification reminders (UI-specific)

When asking the user for input during /planUI, classify:

- **MISSING_INFO**: "I need the brand URL / target audience / primary verb before I proceed" — blocks
- **AMBIGUOUS**: "Is this a marketing landing page or an in-app dashboard?" — binary route choice
- **APPROACH_CHOICE**: "I can go brand-first (Hue-generated) or system-first (ui-design-system). Brand-first is more opinionated; system-first is more flexible. Which?"
- **RISK_CONFIRM**: "Adding 3 canvas backgrounds on the hero will drop Lighthouse perf below 60. Proceed?"
- **SUGGESTION**: "Consider using `hue` for this — the brand signals suggest a brand-first flow"

## Guardrails

1. **Don't spray components** — pick the minimum. One hero background, one headline treatment, one CTA effect. More = noise.
2. **Don't sacrifice a11y for aesthetics** — if a react-bits component has no ARIA, don't use it for primary navigation.
3. **Don't stack heavy canvas backgrounds** — each one costs GPU budget. Profile before stacking.
4. **Don't skip the audit** — impeccable-audit catches the things humans miss. Run it.
5. **Don't guess at brand** — if the user didn't specify, ASK. Defaulting produces generic AI-slop.
6. **Reference, don't copy** — the 17 Hue examples are inspiration, not drop-in templates.

## Hard rules (inherited from /plan)

1. Understand before building — read existing `system.md`, CLAUDE.md, project patterns first.
2. Match capabilities — pick the RIGHT skill for the task. Don't duplicate work across skills.
3. Parallelize independent work (max 3 background subagents).
4. Test everything — visual proof, Lighthouse, a11y audit, responsive screenshots.
5. Simplicity compounds — removing a needless animated background beats adding one.
6. Evidence, not assertions — show the screenshots; show the Lighthouse output.
7. Never break main — always work on a feature branch.
8. Demand elegance — "is there a simpler way?" before accepting any solution.
9. Deliver complete work — don't stop at "here's the plan" unless asked.
10. Route before burning tokens — for scaffold/explore/cherry-pick, dispatch to Qwen; for orchestration + governance, stay on Claude Code.

## Success criteria

A `/planUI` run is complete when:

- [ ] The design brief is answered in concrete terms (who / what / feel)
- [ ] Direction is stated (palette / depth / surfaces / typography / spacing / motion) with WHY
- [ ] Tokens exist (either Hue-generated, ui-design-system-generated, or inferred and saved to `system.md`)
- [ ] Components are built using the minimum set
- [ ] Audit passed (impeccable-audit score ≥ threshold; a11y clean; responsive verified)
- [ ] Polish pass ran
- [ ] Evidence collected (screenshots, Lighthouse, a11y output)
- [ ] Patterns saved back to `system.md` if new ones emerged
